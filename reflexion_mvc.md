5.	¿Qué responsabilidad asume cada uno de los tres componentes del MVC en esta aplicación?
- Cada componente asume responsabilidades distintas. En este caso, el modelo (models) se encarga de gestionar los datos y la lógica relacionada con los productos, como el almacenamiento de datos, listarlos y agregarlos. Por otro lado, las vistas (templates) se encargan de la interfaz de usuario utilizando plantillas HTML para mostrar la información de manera visual. Finalmente, el controlador (controllers) se encarga de recibir las solicitudes del usuario, coordinar la comunicación entre el modelo y la vista, y decidir qué respuesta devolver.
6.	¿Qué ventajas aporta el uso de Blueprints frente a registrar todas las rutas en app.py?
- El uso de Blueprints ayuda a organizar mejor el proyecto, ya que separa las rutas según su funcionalidad o módulo. De esta manera, el código se vuelve más limpio y fácil de comprender para otras personas, especialmente en trabajos en equipo. Además, mejora la reutilización del código y permite registrar grupos de rutas de forma más ordenada dentro de la aplicación Flask.
7.	¿Dónde queda mezclado el acceso a datos con la lógica del modelo? ¿Por qué podría ser un problema?
- El acceso a los datos se encuentra combinado con la lógica del modelo en el archivo producto.py, dentro de la clase Producto, porque en ella no solo se definen las operaciones relacionadas con los productos, como agregarlos o listarlos, sino que también se almacenan directamente los datos mediante la lista _productos. Esto puede generar inconvenientes, ya que el modelo depende del mismo mecanismo de almacenamiento. Por ello, si más adelante se quisiera reemplazar la lista en memoria por una base de datos, habría que realizar cambios importantes en el código del modelo, haciendo el sistema más difícil de mantener y adaptar.
